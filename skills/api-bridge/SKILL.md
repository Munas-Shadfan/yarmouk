# API Bridge

This skill guides you through creating and maintaining a "bridge" between a
frontend agent (typically Angular/TypeScript) and a backend service (commonly
ASP.NET/.NET) so both sides can communicate reliably.  The core idea is to treat
an OpenAPI/Swagger specification stored in a shared folder as the single source
of truth.  The frontend generates clients from the spec and the backend
exports/validates against it.  A small "integration agent" can automate
regeneration on changes.

## What this skill does

- explains why a contract-first approach avoids mismatches
- suggests tooling and existing skills (openapi-spec-generation, orval,
  openapi-to-typescript, etc.) to generate specs and clients
- outlines a shared folder/docs workflow that both agents watch
- provides templates and code snippets for request/response, error handling,
  versioning, auth, and testing
- recommends Angular best practices for consuming the generated API
- optionally shows how to build a helper skill that re‑runs generators when the
  spec changes

## When to trigger

Use this skill when the user:

1. Mentions OpenAPI/Swagger, spec, contract, or shared API documentation.
2. Asks for a TypeScript/Angular client generator for a backend API.
3. Wants to "bridge" or "sync" frontend and backend, or make the frontend
   "understand" the backend.
4. Is exploring tooling like Orval, OpenAPI generation, or .NET API docs.
5. Talks about keeping frontend and backend agents in communication via files,
   shared notes, or a contract.

If the user only asks for Angular design guidelines without API context, do not
trigger.

## Workflow outline

1. **Backend exports spec.**  You need an OpenAPI/Swagger JSON file that
   describes your endpoints.  This can be generated by a .NET tool (Swashbuckle,
   NSwag, etc.) or by one of the OpenAPI generator skills, but you do **not**
   have to install any additional agent skill if you prefer manual commands.

   Example using the bundled script (no other skill required):
   ```bash
   cd /path/to/YourWebProject
   ../skills/api-bridge/scripts/generate_spec.sh YourProject.csproj \
     ../shared/api/openapi.json
   ```
   That script adds Swashbuckle, runs the project on a temporary port and
   curls `/swagger/v1/swagger.json` into the shared folder.  You can adapt it
   for NSwag or another approach.

   Alternatively you could rely on an existing skill like:
   ```bash
   npx skills run wshobson/agents@openapi-spec-generation \
     --output shared/api/openapi.json
   # or npx skills run github/awesome-copilot@aspnet-minimal-api-openapi
   ```

   Commit the generated spec to `shared/api/` (or whichever folder the agents
   share) so both frontend and backend see the same file.

2. **Frontend consumes spec.**  Generate a TypeScript/Angular client from
   the shared OpenAPI file.  You can call an npm package directly, use the
   bundled helper script, or still lean on an agent skill if you like – the
   `api-bridge` skill itself only points you to the right commands.

   Bundled script (no other skill needed):
   ```bash
   cd /path/to/angular-app
   ../skills/api-bridge/scripts/generate_client.sh \
     ../shared/api/openapi.json src/app/generated-api
   ```
   That script will use `orval` if it’s installed, or fall back to
   `openapi-generator-cli` (installable via `npm i -g openapi-generator-cli`).
   Either tool produces strongly‑typed services you can import like this:

   ```ts
   import { ApiService } from './generated-api';
   ```

   If you prefer to keep using an agent skill there’s still the option:
   ```bash
   npx skills run softaworks/agent-toolkit@openapi-to-typescript \
     --input ../shared/api/openapi.json --output src/app/api.ts
   # or npx skills run orval-labs/orval@orval --input ../shared/api/openapi.json
   ```
   but the core workflow doesn’t depend on those packages being installed.

3. **Integration guidance.**  Use the `api integration specialist` skill or the
   snippets below to handle:
   - HTTP errors & retries
   - auth headers / JWT refresh
   - fallback mocks for offline testing
   - API versioning strategies
   - running `ng test`/`dotnet test` when spec changes

4. **Automation.**  Optionally create a tiny watcher skill or CI job that runs
   whenever either side detects a change to `shared/api/openapi.json`.
   Example skeleton:
   ```bash
   # skills/api-bridge/scripts/watch-and-regenerate.sh
   while inotifywait -e close_write shared/api/openapi.json; do
     echo "spec changed, regenerating clients..."
     npx skills run softaworks/agent-toolkit@openapi-to-typescript ...
   done
   ```
   You can wrap this in a skill to run on either agent.

5. **Documentation & sync.**  Publish `shared/api/index.md` with links and a
   changelog so developers (and agents) can quickly see the current contract.

## Code snippets

```ts
// angular service using generated client
import { ApiService } from './api';

@Injectable({ providedIn: 'root' })
export class UsersFacade {
  constructor(private api: ApiService) {}

  listUsers() {
    return this.api.getUsers().pipe(
      catchError(err => {
        console.error('api error', err);
        return of([]);
      })
    );
  }
}
```

```csharp
// .NET controller that validates against spec (pseudo)
[ApiController]
[Route("api/[controller]")]
public class UsersController : ControllerBase
{
    [HttpGet]
    public IEnumerable<User> GetUsers() => _db.Users;
}
```
